## Summary

Adds a canonical internal threat model for `quickjs-wasm` and updates the README with a prominent security warning and a dedicated security guidance section. The docs now clearly describe trust boundaries, supply-chain context, and the in-process execution model so deployers do not treat this package as a memory-isolation sandbox.

## Changes

### Threat Model (`.github/THREAT_MODEL.md`)
- Added a full threat model covering:
  - scope, assumptions, architecture, components, trust boundaries, and data flows
  - threat inventory with severity/status/validation and investigated-dismissed findings
  - input-source coverage and out-of-scope rationale
- Added a supply-chain addendum that documents:
  - dependency topology (`pyo3`, `rquickjs`, `rquickjs-sys`, `oxidase`)
  - `quickjs-ng` as a critical transitive runtime dependency via `rquickjs-sys`
  - maintenance/adoption context and memory-sandbox boundary clarifications
- Added table of contents and Mermaid architecture diagram for readability.

### README Security Addendum (`README.md`)
- Added a top-level `WARNING` callout linking to a new Security section.
- Added a `## Security` section that calls out:
  - this library is not a host-memory isolation boundary
  - quickjs-ng executes in-process with Python
  - guidance for isolated worker/container execution for untrusted code
  - host callback capability risk and least-privilege expectations
- Linked readers to `.github/THREAT_MODEL.md` for full analysis.
